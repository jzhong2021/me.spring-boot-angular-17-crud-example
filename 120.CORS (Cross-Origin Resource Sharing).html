Question 1

Browsers enforce CORS to prevent malicious websites from accessing resources on another domain without permission. Even though both servers are on localhost, the ports (4200 vs 8080) make them different origins.
Exactly right ✅

When it comes to CORS (Cross-Origin Resource Sharing), the concept of an “origin” is defined very strictly — it’s the combination of:

scheme (protocol) + hostname + port

So, for example:
- http://localhost:4200
- http://localhost:8080

Even though both are on localhost, they’re considered different origins because the ports differ (4200 ≠ 8080).

Browsers enforce this separation to prevent a web app from one origin from reading or manipulating data from another origin without explicit permission — a core part of the same-origin policy that helps stop attacks like CSRF and data theft.

That’s why, in development, when your frontend (e.g., Angular on port 4200) calls your backend (on port 8080), you get CORS errors unless the backend explicitly allows that origin, typically via headers like:

Access-Control-Allow-Origin: http://localhost:4200
Access-Control-Allow-Methods: GET, POST, PUT, DELETE
Access-Control-Allow-Headers: Content-Type, Authorization

在浏览器的安全机制中，CORS（跨域资源共享） 是为了防止恶意网站在未经授权的情况下访问另一个域的资源。

一个“源（origin）”由以下三部分组成：

协议（scheme） + 主机名（hostname） + 端口号（port）

示例：
- http://localhost:4200
- http://localhost:8080

虽然它们都在 localhost，但因为端口号不同，所以属于不同的源。

浏览器强制执行这种同源策略，目的是防止一个网页随意访问另一个来源的数据，避免 CSRF 及敏感信息泄露等安全问题。

因此前端（如 Angular 的 4200 端口）访问后端（8080 端口）时会出现 CORS 错误，需要在后端显式允许跨域请求，例如：

Access-Control-Allow-Origin: http://localhost:4200
Access-Control-Allow-Methods: GET, POST, PUT, DELETE
Access-Control-Allow-Headers: Content-Type, Authorization
